Supabase-shaped chainable builder: `select()`, `insert()`, `upsert()`, `update()`, `delete()`, `eq/neq/gt/gte/lt/lte/is/not/contains/ilike/in()`, `order()`, `limit()`, `single()`, `maybeSingle()`, thenable (`.then()`), plus `or()` and `range()` which are stubs that throw. **JOIN_CONFIG** currently contains exactly **one** entry: ```js "comments.profiles": { localField: "user_id", targetCollection: "profiles", targetField: "shared_user_id" } ``` Any embedded-select relation not listed here (`"&lt;collection&gt;.&lt;embedName&gt;"`) throws in `parseSelect()`: `Unsupported embedded select "X" on "Y" — not yet configured in JOIN_CONFIG (real-query.ts). This route hasn't been ported off the old stub client yet.` Embeds are resolved with a second `pbList` call using `field = 'v1' || field = 'v2' ...` (batched, not a real SQL join — PocketBase filter syntax). **`.eq()` / filters**: build a PocketBase filter-string expression (`col = 'value'`), joined with `&amp;&amp;`. `.in()` builds an OR-group of `=` comparisons; empty array short-circuits to an always-false filter. **`.or()`**: `throw new Error(\`RealPocketBaseQuery.or() is not implemented yet (collection: ${this.collection}). Not ported.\`)` — immediate throw, not silent no-op. **`.range()`**: same pattern — `throw new Error(\`RealPocketBaseQuery.range() is not implemented yet...\`)`. Both throws are caught by the top-level `execute()` try/catch and surfaced as `{ data: null, error: { message } }`, so callers see a `400`-style error response rather than a crash — but functionally the feature silently fails from the user's perspective (e.g., pagination or `.or()` search just errors out). Old/stub client — `src/lib/pocketbase/client.ts` (199 lines, full read) This is the **browser-side** stub (`"use client"` at top), exported as `createClient()`. It is NOT what most server routes use anymore (see below) — it's still live and used by client components. - `.from(collection)` returns a `PocketBaseQuery` class where every filter method (`eq/neq/gt/...in/order/limit/range/or`) is a no-op that just `return this`. - `.then()` always resolves to `{ data: [], error: null, count: 0 }` — i.e. every select silently returns empty data. - `.single()` has one hardcoded special case: `profiles` + selecting `data_chips` proxies to `GET /api/saily/chips/balance`; everything else resolves `{ data: null, error: null }`. - `.maybeSingle()` always resolves `{ data: null, error: null }`. - `insert/upsert/update/delete` are no-ops returning `this`. - `.rpc(fn)`: only `"unlock_archive"` is special-cased (proxies to `/api/saily/chips/unlock-archive`); everything else resolves `{ data: null, error: null }`. - Real auth (`signInWithPassword`, `signUp`, `signOut`, session read/write via localStorage + cookie) is implemented for real — only the `.from()` query builder is a stub. **Important architectural note:** `src/lib/pocketbase/server.ts` (the server-side `createClient()`, used by nearly every API route and server component) **already wires `.from()` to `RealPocketBaseQuery` unconditionally** — it is not a toggle/stub anymore. So "porting a route off the stub" for server routes doesn't mean swapping an import; it means fixing/extending `real-query.ts` (JOIN_CONFIG, `.or()`, `.range()`) so the route's *existing* calls stop throwing at runtime. The stub `client.ts` is only still truly "live" for **client components** (`"use client"` files), which cannot safely import `real-query.ts` since it needs the PocketBase superuser token (`admin.ts`) — a server-only secret. `server.ts` also documents `UNIMPLEMENTED_RPCS = new Set(["award_comment_badges", "ensure_forum_threads", "submit_daily_result"])` — calling any of these logs a warning and no-ops, silently (`rpc()` doesn't throw for these). ## 3.
