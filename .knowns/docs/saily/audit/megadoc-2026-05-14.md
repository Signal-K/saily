## Section 1: Project Overview

### What Saily Is

Saily (branded "The Daily Sail") is a Wordle-style daily citizen science web app. Users classify real astronomical and planetary imagery — TESS lightcurves, Mars rover photographs, asteroid stamp images — inside a lightweight narrative frame. One mission per day, resetting at Melbourne/AEST midnight. The product tagline is "grounded daily citizen science puzzles."

Each day, a "mission" presents three mini-games in sequence:
1. **Planet Hunter** — annotate transit dips on a TESS lightcurve
2. **Asteroid Survey** — tag anomalies on an asteroid image stamp
3. **Mars Surface Survey** — classify surface features on a NASA rover photograph

Between games, a character (currently "Gizmo") delivers 1–2-sentence narrative updates. The arc-level structure (5 chapters per storyline) tracks per-user progress with chapter advancement on completion.

All four science feeds have:

**None of the cache tables contain real data.** The fallback subjects are placeholders with Zooniverse background image URLs and synthetic data.

| Cloudspotting on Mars | `cloudspotting_mars_daily` | `ingest-cloudspotting-mars.mjs` | `/api/cloudspotting-mars/daily` → Cloudspotting game | 3 NASA photojournal images (real, public domain) |

The **asteroid game currently does not use its cache table**. `AsteroidGamePage` reads from `ASTEROID_ANOMALIES` in `secondary-game.ts`, which contains 3 hardcoded TESS stubs with `imagePath: "/puzzles/puzzle-*.svg"` — SVG placeholder files. The `active_asteroids_daily` table exists but nothing reads from it during gameplay.

The **Rubin and Gaia feeds have no game UI at all** — the cache tables, ingestion scripts, and TypeScript types exist, but there is no component or page that renders a Rubin or Gaia mission.

### Mars Game — Live API Dependency (Architectural Risk)

`/api/mars/daily` fetches live from `https://images-api.nasa.gov/search` at request time with no caching layer. If the NASA API is slow or down, the Mars game falls back to a static set of 15 images. This is an architectural violation of the spec's own principle: "Do not rely on live third-party APIs directly in request-time gameplay." The Cloudspotting on Mars feed correctly uses a cache table; Mars imagery should be migrated to the same pattern (the `mars_images_pool` table exists from migration `20260408151000`).

### Database Design Issues

1. **`anomalies` table name mismatch:** The table uses camelCase column names (`ticId`, `anomalySet`, `anomalyConfiguration`) in a PostgreSQL schema, which requires quoting in every query. This is an ongoing friction point visible in `api/game/today/route.ts` where columns are quoted individually.

2. **`repair_streak` inserts `attempts=0`:** The `repair_streak` RPC inserts a play record with `attempts=0, won=true, is_repaired=true`. The `daily_plays` table has a CHECK constraint: `attempts >= 1 AND attempts <= 10`. **This is a bug** — the RPC would fail at runtime for the `repair_streak` path. If Supabase is running with the migration applied, attempting to repair a streak will throw a constraint violation.

3. **`user_story_progress` has no completed flag:** Chapter advancement is tracked by incrementing `chapter_index`. There is no explicit `completed_at` or `is_complete` column. Completion is inferred by `chapter_index >= storyline.chapters.length`. If chapter count changes, existing "complete" users may find themselves mid-arc again.

4. **Science feed tables have no RLS INSERT policies:** `cloudspotting_mars_daily` and siblings allow SELECT by anyone but have no INSERT policy. Ingestion scripts must use the `service_role` key — this is correct design but it means the scripts will silently fail if run with the anon key.

### Auth Setup

Magic-link auth via Supabase GoTrue. No social auth in production (`NEXT_PUBLIC_AUTH_GOOGLE_ENABLED: "false"`). The `handle_new_user` trigger (migration `20260508110000`) handles username collision by appending `_N` suffixes, generates a referral code, and awards a bonus chip to the referrer. This is robust. The docker-compose JWT secret and anon/service keys are demo values — these must be rotated before any external deployment.

### Missing Infrastructure

- No edge function or cron for science data refresh. The research doc says "daily cache table intake" but no automation exists. Ingestion must be run manually: `node scripts/ingest-cloudspotting-mars.mjs`.
- No production deployment config is present in the repo (no `vercel.json`, no env file template). The v0 readiness script assumes Vercel but the deployment target is undocumented.
- No CORS configuration beyond the Docker Kong setup.
- Cypress tests require `E2E_TEST_EMAIL` and `E2E_TEST_PASSWORD` env vars — these are not documented in any `.env.example`.

### Data Quality Concerns

**Cloudspotting on Mars fallback:** 3 real NASA images (Odyssey, InSight, Pathfinder). These are genuine Mars cloud photographs. The fallback is reasonable for demos but scientifically thin — cloud classification needs a structured template dictionary (like the Mars surface dictionary exists in `mars-dictionary.ts`).

**Active Asteroids fallback:** The single fallback subject points to `https://www.zooniverse.org/assets/project-background.jpg` — this is a generic Zooniverse background image, not an asteroid stamp. Any user hitting this fallback sees an incorrect image.

**Rubin Comet Catchers fallback:** Same problem — `https://www.zooniverse.org/assets/project-background.jpg` placeholder.

**Gaia Variables fallback:** A 3-point synthetic time series `[{x:0, y:1}, {x:0.5, y:0.5}, {x:1, y:1}]`. This is trivially simple and bears no resemblance to real variable-star data. The game component for Gaia doesn't exist yet, so this doesn't affect users today.

**Mars live API:** The `/api/mars/daily` route fetches from `https://images-api.nasa.gov/search` at request time, filtering results by `isLikelyRoverPhoto()`. If NASA API returns no qualifying images for a given page, the fallback is 15 static images. This is the only game currently operating at real-world fidelity, but its request-time dependency is a stability risk.

**TESS lightcurve (planet game):** The `anomalies` table contains real TESS data (CSV-seeded). `generateSyntheticLightcurve()` exists as a fallback and is used when the anomaly count is 0. The API selects up to 24 anomalies starting from a date-seeded offset and ranks them by signal strength to show the top 3. This is the most mature data pipeline.

### Data Pipeline Gaps

1. **No automation for feed refresh.** All ingestion scripts must be run manually. No cron, no Supabase Edge Function, no CI/CD hook.
2. **No data validation on ingest.** Scripts insert records but don't verify that `image_url` values are reachable.
3. **No content moderation layer.** Images come from Zooniverse subject sets. If a subject set changes or an image URL rotates, the cache will serve stale or broken URLs.
4. **No deduplication guard on daily subject selection.** The same subject could theoretically appear on multiple consecutive dates if the date-seeded offset lands on the same row.

**H3 — Hint penalty scoring not implemented**
- Product spec and system spec both describe a hint penalty (0.75x for Phase-Fold/Binning use). This is specified in FR-2 and the reward multiplier section. The planet game (`TodayGamePage`) has Phase-Fold and Binning tool controls, but hint usage is never sent to the scoring API.

- `/api/mars/daily` makes a real HTTP request to `https://images-api.nasa.gov/search` for every page load.

- The `mars_images_pool` table exists (migration `20260408151000`) for pre-caching but is not used by the Mars game.
- Fix: Migrate Mars game to use the `mars_images_pool` cache table and add a daily ingestion script.

**H5 — Docker JWT secrets are demo values**
- `docker-compose.yml` contains hardcoded demo anon key, service key, and JWT secret (`super-secret-jwt-token-with-at-least-32-characters-long`).
- These are standard Supabase local dev demo values but they must never reach production. There is no `.env.example` documenting what secrets to replace.
- Fix: Add `.env.example` with all required secrets, document the secret rotation step in the setup guide.

**H6 — No ingestion automation for any science feed**
- All four science feed cache tables are empty. There is no scheduled job, cron, or CI pipeline to populate them. The product depends on a developer manually running `node scripts/ingest-*.mjs` before any real science data appears.
- Fix: Add at least one Edge Function or a GitHub Actions cron to run daily ingestion.

**H7 — `game.ts` word puzzle is dead code**
- `web/src/lib/game.ts` contains a full word-unscramble puzzle engine (`WORD_BANK`, `getPuzzleForDate`, `scoreGuess`). The `/api/game/today` route returns a `puzzle` field using this engine. No page renders this word puzzle — `MissionFlowPage` routes entirely to the lightcurve/asteroid/mars games.
- Fix: Remove the word puzzle engine and `puzzle` field from the API response, or document it as a planned feature.

**H8 — Rubin Comet Catchers and Gaia Variables have no game UI**
- Both feeds have cache tables, TypeScript types, and ingestion scripts, but there is no game component or page that renders them. A player has no path to experience these science feeds.
- Fix: Either build game components for each feed or document them as post-MVP and remove the cache tables from the current sprint scope.

- Fix: Audit `/api/forum/threads` and `/api/comments` to confirm mission completion is checked before allowing reads/writes.

**M4 — Username collision loop is unbounded**
- Migration `20260508110000` uses a `while exists` loop to find a unique username. For a large user base with common email prefixes (e.g., `john@*`), this loop runs until a suffix is found. There is no max-iterations guard.
- Fix: Add `AND v_suffix < 1000` guard and fallback to UUID suffix.

**M5 — Docker `make up` requires Supabase CLI already running**
- The Makefile's `up` target calls `check-supabase` which verifies the Supabase CLI is running locally, but the Docker Compose file has its own Supabase services. These are two different Supabase instances (CLI vs Docker-native). The `make up` flow is inconsistent — it was originally designed around CLI + Docker-web hybrid, and the containerized Supabase was added later without fully resolving the conflict.
- Fix: Remove `check-supabase` from the `up` target since Docker Compose brings up its own Supabase services.

**M6 — No `.env.docker` file documented**
- `make up` passes `--env-file .env.docker` but no `.env.docker` file or template exists in the repo.
- Fix: Add `.env.docker.example`.

**M7 — `completed_storylines` array could accumulate stale IDs**
- If a storyline ID changes (e.g., from `"gizmo"` to something else in a future refactor), old `completed_storylines` values in `profiles` will contain the old ID. The double-reward check in `story/progress/route.ts` would then fail silently and award chips again.
- Fix: Either use a separate `user_storyline_completions` table with a foreign-key-like structure, or document the migration path for storyline ID changes.
